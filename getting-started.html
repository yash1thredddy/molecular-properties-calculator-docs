Getting started

A typical session, from a single molecule to a whole library — no coding, no setup, just paste or upload and read the results.

Before you begin: nothing to install

The Molecular Properties Calculator runs entirely in your web browser. There is nothing to download, no account to create, and no programming required. Click Open the App ↗ and you are ready to go.

Inside the app, the very first choice you make is which workspace to use. A workspace (the app also calls these "modes") is just a screen tuned to one kind of question — looking at a single compound, crunching a whole spreadsheet, drawing charts, fitting a trend, or finding hidden groups. You pick one, do your work, and switch freely between them.

A quick vocabulary note

Throughout this guide, a SMILES is a short text recipe for a molecule (for example, aspirin is CC(=O)OC1=CC=CC=C1C(=O)O). A property is a single calculated number describing the molecule (its weight, its greasiness, and so on). A workspace is one of the app's five screens. Every other term is defined the first time it appears.

The five workspaces — and the question each one answers

Each workspace is built around a different question. Start from the one that matches what you want to learn. Every card below links to that workspace's dedicated page in this guide, where every control is explained.

Single Molecule

"What is this one compound like?" Paste a single structure and get its full property breakdown, interference flags, and a 2D picture of the molecule.

Batch Processing

"Can I calculate everything for my whole list at once?" Upload a spreadsheet of thousands of molecules and compute properties and efficiency indices for all of them in one pass.

Data Visualization

"How do these properties relate to each other?" Turn a dataset into interactive charts — scatter plots, histograms, heatmaps — to spot trends and outliers. Click any point to see that molecule.

3D Regression

"Does property A and B predict property C?" Fit a flat plane through three numeric properties to study structure–activity relationships, with full statistics.

GMM Analysis

"Are there natural sub-groups hiding in my data?" Let the tool discover clusters — like "small & polar" versus "large & greasy" — automatically, without you defining the groups in advance. (GMM stands for Gaussian Mixture Model; the GMM page explains it in plain words.)

Not sure which to pick?

If you have one compound, start with Single Molecule. If you have a spreadsheet, start with Batch Processing to get all the numbers, then move into Data Visualization, 3D Regression, or GMM to explore them.

The end-to-end workflow

No matter which workspace you choose, the rhythm is the same five steps. Once you have done it once, every workspace will feel familiar.

1 · Pick a workspace

Choose the mode that matches your question (the five cards above). You can switch at any time — your uploaded data stays available while you explore.

2 · Provide your molecule (or molecules)

For a single compound, type or paste its structure into the input box — a SMILES, an InChI, or an InChI Key all work (see input formats below for what each is). For many compounds, upload a file: a CSV or XLSX spreadsheet with one molecule per row. The tool automatically finds the column holding the structures — it recognizes common names like SMILES, smiles, SMI, or canonical_smiles.

3 · Choose what to calculate

Tick the properties or analyses you want — all 28 properties, only the drug-likeness rules, the ligand-efficiency indices, the interference flags, and so on. You only compute what you need, which keeps things fast. (Each item is explained on its own page under The Numbers in the sidebar.)

4 · Read the results

The tool shows your answers as tables of numbers, interactive charts, and — for single molecules — a 2D structure drawing of the compound. Hover, zoom, and (in visualization mode) click any point to pop up that molecule's picture. Invalid or unreadable structures are reported clearly rather than silently dropped.

5 · Download your enriched data

Export the results as a CSV or XLSX file. The output is clean, numeric columns that open directly in Excel and load cleanly into downstream tools like KNIME, StarDrop, and Pipeline Pilot for further work.

One file at a time

Each workspace works with one uploaded file at a time. When you upload a new file, it replaces the previous one — there is no hidden "merge." If you want to combine datasets, join them in your spreadsheet program first, then upload the single combined file.

One reference page, always handy

The Reference page answers the most common questions and defines every term used across this guide in plain words. Keep it open in a tab as you work.

What you type in: input formats

A molecule must be written so a computer can read it. There are three standard formats; you only need one per molecule.

Why a molecule needs to be "written down"

A drawing of a molecule is easy for a chemist to read, but a computer can't work with a picture. It needs the structure expressed as plain text — letters, numbers, and symbols on a single line — so it can parse the atoms, the bonds between them, and how everything is arranged. That line of text is what you paste into the app, or what sits in a column of your spreadsheet.

The good news: you don't have to learn how to write these strings by hand. You'll usually copy them from a database (PubChem, ChEMBL), export them from drawing software (ChemDraw, MarvinSketch), or already have them in a file. This page explains what each one is, so you can recognize them and pick the right one.

The one rule to remember

You only need one format per molecule. SMILES, InChI, and InChI Key are three different ways of writing the same structure. Choose whichever you happen to have — the app figures out the rest.

SMILES — the molecule as a short recipe

SMILES (Simplified Molecular-Input Line-Entry System) writes a molecule as a short line of text — a bit like a recipe that walks through the atoms in order. Letters are atoms (C for carbon, O for oxygen, N for nitrogen), numbers mark where a ring closes, parentheses hold branches, and symbols like = mark double bonds.

Aspirin in SMILES is CC(=O)OC1=CC=CC=C1C(=O)O. You can read it loosely as: a carbon, then a carbon carrying a double-bonded oxygen, then an oxygen, then a six-membered ring (the 1 opens and later closes the ring), and a final acid group.

Short and readable Works fully offline The app's preferred format
One molecule can have several valid SMILES

SMILES is flexible — the same molecule can be written starting from different atoms, so two correct SMILES strings may look different yet mean the same thing. That flexibility is why SMILES is easy for people, and also why the stricter InChI (below) exists when you need one guaranteed-identical text per structure.

InChI — the molecule as a strict barcode

InChI (International Chemical Identifier) is a longer, stricter way of writing a structure. Think of it as a barcode: it follows rigid rules so that the same molecule always produces the exact same InChI string, no matter who generates it or which software they use. That makes it perfect for matching records across databases and for checking whether two entries are really the same compound.

Aspirin's InChI begins InChI=1S/C9H8O4/c1-6(10)13-8-5-3-2-4-7(8)9(11)12/…. The 1S means "standard version," C9H8O4 is the formula, and the parts after the slashes encode how the atoms connect and where the hydrogens and charges sit. You never need to read this — you just need to know it's unique and reproducible.

Same molecule → same InChI, every time Great for database matching Works offline in the app

InChI Key — a short fingerprint of the InChI

An InChI string can get long. The InChI Key squeezes it down to a short, fixed-length code — always 27 characters in three blocks, like BSYNRYMUTXBXSQ-UHFFFAOYSA-N for aspirin. It's a fingerprint: a compact stand-in that's convenient to store, search, and paste into a search box. The first block captures the skeleton; the rest captures finer details and a final letter for charge.

The catch — and it's an important one — is that an InChI Key is a one-way fingerprint. Just as you can't reconstruct a person from their fingerprint, you can't rebuild the full structure from the key alone. The key is derived from the structure, but the math only runs in that one direction.

Why InChI Keys need the internet

Because the key is one-way, the app can't turn it back into a structure by itself. Instead it looks the key up in an online chemistry database to fetch the real structure: it first asks the NIH CIR resolver (cactus.nci.nih.gov), and if that doesn't answer it falls back to PubChem. Once the structure comes back, it's processed exactly like a SMILES. These lookups are rate-limited and cached so repeated keys stay fast, but they do require a working connection — and a key that isn't in either database simply can't be resolved.

Prefer SMILES whenever you can

SMILES and InChI both work without a connection, but SMILES is the shortest and the format the app is happiest with. If you want everything to run fully offline — and to avoid any dependence on an external database being available — use SMILES. Reach for InChI Key only when that's the only identifier you have.

Uploading a file: how the structure column is found

When you upload a spreadsheet — a .csv or .xlsx file — for batch processing or visualization, the app needs to know which column holds your structures. It detects this automatically by looking at your column headers and matching common names for a structure column.

If one of your columns is named any of the following (capitalization doesn't matter), the app picks it up as the structure column:

SMILES smiles SMI canonical_smiles

💡

What this shows: the app scans your headers, spots the SMILES column, and reads structures from it — other columns (Name, Activity, Source) are carried along untouched.

If your column has a different name

Name the column SMILES before uploading and detection just works. One file is handled per workspace at a time, and a new upload replaces the previous one. Rows with an invalid or unreadable structure are reported individually — a single bad row won't stop the rest of the file from processing.

Which format should I use?

Have a structure to type?

Use SMILES. It's the shortest, easiest to paste, and runs fully offline.

Matching across databases?

Use InChI when you need one guaranteed-identical text per structure. Also offline.

Only have a key?

Use the InChI Key — just remember it needs the internet to be resolved into a structure.

Common questions

Do I ever need more than one format for a molecule?

No. One format per molecule is enough. The three formats are interchangeable descriptions of the same structure, so you give the app whichever one you have.

My InChI Key won't resolve — what's wrong?

Two usual causes. First, you need a live internet connection — the key is looked up online via NIH CIR (with PubChem as backup). Second, the compound may not exist in either database; brand-new or proprietary structures often aren't there. If you have the SMILES or InChI, use that instead and the lookup is skipped entirely.

Why does the same molecule sometimes have two different SMILES?

SMILES lets you start the "recipe" from different atoms, so several correct strings can describe one structure. They're all valid and the app reads them identically. If you need a single canonical text per molecule for matching, use InChI instead.

What file types can I upload, and how is the structure column found?

.csv and .xlsx files. The app auto-detects the structure column from its header name — SMILES, smiles, SMI, or canonical_smiles. Name your column one of those and it's recognized automatically.
